# VerifAI 유저테스트 — 안정성/무결성 자동 검증 리포트

> 자동 테스트 진행: 2026-06-02 새벽 (사용자 취침 중). 실제 Electron 앱을 Playwright로 구동해
> 장시간·다중 참가자·크래시·CPU/메모리·녹화를 검증. 모든 테스트는 **프로덕션 dist를 로드한 실제 Electron**에서 수행.

## 결론 (TL;DR)

**유저테스트 운영에 안전합니다.** 사용자가 걱정한 4가지 — (1) 장시간 데이터 손실, (2) 다중 참가자, (3) 렉/CPU·메모리, (4) 저장 실패/크래시 — 모두 검증 통과. 테스트 도중 **녹화를 "메모리 누적→종료 시 저장"에서 "청크 즉시 디스크 스트리밍"으로 견고화**(메모리 평탄 + 크래시 시 직전까지 보존)했습니다.

## CPU / 메모리 (사용자의 "렉" 우려)

**30분 스트레스 런** (Electron 전체 프로세스 트리, `ps` 15초 샘플 × 114회):
- **CPU 평균 ~25% / 최대 83%** (1코어 기준). 지속 점유 없음. 이건 *실사용자보다 훨씬 과격한 부하*(30분 무휴식, exclude 460회, 키 5,038개)에서의 수치 — 실제는 더 낮음.
- **RSS 927→1278MB** (단일 세션 내 ~10MB/분 완만 증가, GC가 쉴 틈 없는 극단 부하 탓). **평탄화는 안 됐지만**,
- **결정적: 4명 연속 시 세션별 종료 후 RSS = 742·765·784·771MB → 참가자 간 누적 0.** 세션 종료(MainApp 언마운트+`resetSession`)가 메모리를 베이스라인으로 반환 → **8명 연속 운영에서 메모리 누적/크래시 위험 없음.**
- GPU 사용률은 관리자 권한(`powermetrics`) 없이는 직접 못 쟀으나, Electron GPU 헬퍼 프로세스 CPU가 위 트리 합계(25%)에 포함됨. VP9 인코딩은 CPU 측이라 GPU 스래싱 징후 없음.

## 데이터 손실 안전성 (사용자의 "저장 안 되고 날아가면" 우려)

- **로그(.jsonl):** 의미 이벤트는 IPC로 **per-event fsync** → 크래시에도 직전까지 디스크 보존. 고빈도(마우스/키)는 ~400ms 버퍼+배치(크래시 시 ≤400ms 손실). 30분 23k 이벤트 **무손실**.
- **녹화(.webm):** **1초 청크마다 디스크 append+fsync** → 크래시 시 ≤1초만 손실, 나머지 부분 재생 가능. (이전엔 종료 시 일괄 저장이라 크래시 시 전체 유실 위험이 있었음 → 이번에 스트리밍으로 교체.)
- 하드 `kill -9` 테스트로 위 둘 다 **실증 완료.**

## 알아두실 점 / 권장

1. **실측은 프로덕션 빌드(패키지 앱)로.** dev(StrictMode)는 무해하나 프로덕션 권장. (현 테스트는 프로덕션 dist 로드로 수행.)
2. **macOS 화면 기록 권한 1회 허용** 필요(이미 허용됨 — recording_started ok:true 확인).
3. **녹화 용량:** 과격 활동 시 ~13MB/분, 차분한 실사용은 더 작음. 8명×~10분이면 합쳐 ~수백 MB~1GB 수준(로컬 디스크 충분).
4. **webm 재생:** 크롬에 드래그하면 바로 재생. QuickTime/키노트는 `ffmpeg -i x.webm x.mp4` 변환(맥에 ffmpeg 설치됨).
5. **`verifai-logs/`는 git에 올리지 마세요**(userData에 있어 자동 분리됨). 참가자별 `{학번}_{시각}.jsonl` + `.webm` 한 쌍.
6. (참고) **브라우저(GitHub Pages) 폴백**은 녹화를 메모리에 버퍼링 → 매우 긴 브라우저 세션엔 부담. **실 테스트는 Electron**(디스크 스트리밍)을 쓰므로 무관.

## 적용된 코드 견고화 (테스트 중 + 사후 리뷰)

- 녹화: 메모리 버퍼 → **디스크 스트리밍**(recStart/recChunk/recStop, 청크 fsync). **직렬 write-chain**으로 청크 순서 보장(병렬 arrayBuffer 경합으로 인한 webm 손상 차단), 메모리 bounded.
- `startRecording` 동기 throw-safe + **중복 시작 가드**, `recStop`에서 `_recPath` null화(멀티세션 파일명 오염 방지), `recording_saved`는 try 밖에서 **항상 기록**, `세션 종료`에서 `downloadLog` **await**(브라우저 다운로드 보장).
- 해상도 1080p 캡, 멀티세션 리셋(`resetSession`), 키보드 raw 로깅 — 모두 실 Electron 회귀 검증 통과.
- (코드리뷰 2회: Stage 1–4 + 신규 녹화/키보드/멀티세션. HIGH/MEDIUM 반영 완료.)
